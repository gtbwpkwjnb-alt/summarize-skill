## 平台适配 — 自动检测 Agent 平台（v6.6.1+）

> **设计目标**：summarize 作为通用技能，在首次运行时自动检测当前 Agent 平台，适配 L0 全局规则文件路径、项目记忆文件路径、技能目录路径，确保模块3.3(记忆健康度)、模块6(分层写入) 在不同平台正确读写。

### 平台检测表

检测顺序：依次检查以下路径是否存在，首次命中即确定平台。

| 优先级 | 平台 | L0 全局规则路径 | 项目记忆文件 | 技能目录 | 检测方式 |
|:------:|------|-----------------|-------------|---------|---------|
| 1 | **ZCode** | `~/.agents/AGENTS.md` | `./MEMORY.md` | `~/.agents/skills/` | 默认（当前平台） |
| 2 | **CodeBuddy** | `~/.codebuddy/AGENTS.md` | `./CODEBUDDY.md` | `~/.codebuddy/skills/` | 检查 `~/.codebuddy/` 目录存在 |
| 3 | **Claude Code** | `~/.claude/CLAUDE.md` | `./CLAUDE.md` | `~/.claude/skills/` | 检查 `~/.claude/` 目录存在 |
| 4 | **Codex** | `~/.codex/AGENTS.md` | `./MEMORY.md` | `~/.codex/skills/` | 检查 `~/.codex/` 目录存在 |
| 5 | **Reasonix** | `~/.reasonix/AGENTS.md` | `./REASONIX.md` | `~/.reasonix/skills/` | 检查 `~/.reasonix/` 或项目 `reasonix.toml` |

### 平台检测执行流程

```
Step 1: 按优先级依次检查目录存在性
  → CheckDir(~/.codebuddy/) → 命中 → $PLATFORM="codebuddy"
  → CheckDir(~/.claude/)     → 命中 → $PLATFORM="claude"
  → CheckDir(~/.codex/)      → 命中 → $PLATFORM="codex"
  → CheckDir(~/.reasonix/) OR FileExists(./reasonix.toml) → 命中 → $PLATFORM="reasonix"
  → 全部未命中 → $PLATFORM="zcode"（默认）

Step 2: 设定平台相关路径
  → $L0_RULES = 平台对应的 L0 全局规则路径
  → $PROJ_MEMORY = 平台对应的项目记忆文件名
  → $SKILL_BASE = 平台对应的技能目录

Step 3: 检查路径实际可读性
  → Read $L0_RULES → 成功 → 模块3.3 正常
  → Read $L0_RULES → 失败 → 降级 L3（L0规则不可读），使用默认路径作为写入目标
  → 若 $L0_RULES 文件不存在 → 自动创建空文件（带平台标准头部）

Step 4: 适配模块6写入目标
  → 项目级记忆: ./$PROJ_MEMORY（如 ./CODEBUDDY.md、./CLAUDE.md）
  → 全局级记忆: $L0_RULES 同目录下的 memories/ 子目录
  → 技能级: $SKILL_BASE/summarize/references/learned/
```

### 路径变量映射表（模块6写入时使用）

| 变量 | ZCode | CodeBuddy | Claude Code | Codex | Reasonix |
|------|-------|-----------|-------------|-------|----------|
| `$L0_RULES` | `~/.agents/AGENTS.md` | `~/.codebuddy/AGENTS.md` | `~/.claude/CLAUDE.md` | `~/.codex/AGENTS.md` | `~/.reasonix/AGENTS.md` |
| `$L0_MEM_DIR` | `~/.agents/memories/` | `~/.codebuddy/memories/` | `~/.claude/memories/` | `~/.codex/memories/` | `~/.reasonix/memories/` |
| `$PROJ_MEMORY` | `./MEMORY.md` | `./CODEBUDDY.md` | `./CLAUDE.md` | `./MEMORY.md` | `./REASONIX.md` |
| `$SKILL_BASE` | `~/.agents/skills/` | `~/.codebuddy/skills/` | `~/.claude/skills/` | `~/.codex/skills/` | `~/.reasonix/skills/` |

### 跨平台兼容保证

- **降级安全**：L0 规则文件不可读时，模块3.3(记忆健康度)降级但不影响模块1-2核心输出，模块5+6使用默认路径尝试写入
- **写入容错**：目标目录不存在时自动创建（mkdir -p 等效）
- **读取降级**：平台特定文件不可读时，依次尝试其他平台路径作为 fallback
- **首次运行**：自动创建缺失的 L0 规则文件（含平台标准头部注释），确保后续写入正常

## 存储层级 — 4层金字塔（与全局记忆体系对齐）

**基址**：所有 harvest 文件统一存储在 summarize 技能目录下（跟随技能，跨项目共享）：

```
~/.agents/skills/summarize/harvests/   ← $HARVEST_BASE（固定，无需平台检测）
  {project}/                           ← 每个项目一个子目录（温区）
    {session-id}.md                    ← 会话归档（覆盖更新，不加载上下文）
    errors.md                          ← 温区错误（≤30条，项目专属）
  error-ledger.md                      ← 冷区（≤100条，跨项目）
  index.md                             ← 收割索引（同会话只保留最新一条）
  archive/                             ← 归档（显式恢复，永不自动加载）
    {date}-errors-compact.md           ← 明文存档（主力）
    {date}-errors-compact.md.hr        ← headroom hash（可选加速，纯文本文件存 hash 值）
```

**4层金字塔（与 ERROR-LIFECYCLE.md / ~/.agents/AGENTS.md 对齐）**：

| 层级 | 路径 | 上限 | 加载策略 |
|:----:|------|:---:|:--------:|
| 🔥**热区** | `~/.agents/AGENTS.md` P0表（L0全局规则） | ≤5条 | hook 每次 session 自动注入 |
| 🔶**温区** | `$HARVEST_BASE/{project}/errors.md` | ≤30条 | 按需检索 |
| 🔵**冷区** | `$HARVEST_BASE/error-ledger.md` | ≤100条 | 总结时 + 主动检索 |
| ⬜**归档** | `$HARVEST_BASE/archive/` | 无上限 | 显式恢复，永不自动加载 |

**常驻负担**：仅热区 ~2KB / ~500 token（hook 注入），温冷区按需加载，不对全局记忆形成压力。

**运行时**：`$HARVEST_BASE=~/.agents/skills/summarize/harvests/`（固定路径，无需平台检测）。文件不存在时自动创建。

> **规则源对接**：模块3 收割时加载 L0 `~/.agents/AGENTS.md` 的 P0 表作为规则源（hook 已注入上下文，或主动 Read）。温区错误升级热区时，通过 `总结 深度 写入` 子命令 Edit L0 的 P0 表（受定容5条约束，满则降级最旧条到温区）。

## 边界场景与降级

### 场景处置（自动修复型 — 无需降级即可修复）

> 以下场景可自动修复，不影响模块完整性。不可自动修复的运行时失败走降级链。

| 场景 | 行为 | 输出 |
|------|------|------|
| 基址目录缺失 | 自动创建 `~/.agents/skills/summarize/harvests/` | `ℹ️ 已创建 harvests 目录` |
| 归档文件损坏 | 跳过该文件，重建空文件 | `⚠️ 归档损坏，已重建` |
| 旧版数据格式（v5） | 自动迁移到 v6 结构 | `↻ 数据格式已迁移 v5→v6` |
| 会话 ID 冲突 | 保留新归档，旧归档标 `[superseded]` | `↻ 会话已覆盖旧归档` |
| 温区/冷区文件编码错误 | 检测到非 UTF-8 → 重建空文件，原文件备份为 `.bak` | `⚠️ 编码异常，已重建（原文件→.bak）` |

### 降级链（5级递进 · 按严重程度递增）

> **核心原则**：降级后每次仍**尝试写入**（用于检测恢复条件），失败仅更新计数器不报错。逐级恢复（非跳回L0）。
> **双轨独立降级**：模块5(成功轨) 与模块4(错误轨) 独立降级。模块4降级不影响模块5（成功轨不依赖错误轨写入）。

```
L0 正常（全模块：模块1+2+3+4+5+6）
  │
  ├── 触发条件: 无异常
  │
  ▼
L1 模块4降级（仅内存统计）+ 模块5+6正常
  │  模块1+2+3 正常，模块5+6 正常（成功轨独立运行）
  │  模块4 跳过持久化写入
  │  每次仍尝试写温区/冷区/索引（仅写1次，不重试）→ 成功则计入恢复计数
  │
  ├── 触发条件: 温区 errors.md 写入失败(1次)
  │     OR 冷区 error-ledger.md 写入失败(1次)
  │     OR 索引 index.md 更新失败(1次)
  │     OR 温区/冷区 读取失败（文件损坏/编码错误）→ 触发场景处置重建后仍失败
  │
  ▼
L2 模块3.3+4 跳过（保留模块3.1+3.2）+ 模块5+6正常
  │  模块1+2 正常，模块3.1(链路优化)+3.2(知识沉淀) 正常，模块5+6 正常
  │  模块3.3(记忆健康度) 跳过（依赖 L0 P0表），模块4 跳过
  │  每次仍尝试写温区/冷区/索引 → 成功则计入恢复计数
  │
  ├── 触发条件: 温区 errors.md 写入连续2次失败（L1时累计）
  │     OR 冷区 error-ledger.md 写入连续2次失败（L1时累计）
  │     OR 温区+冷区同时读取失败（文件损坏无法重建）
  │
  ▼
L3 仅模块1+2 + 模块5降为仅内存采集
  │  仅输出项目总结+进度总结，无建议/错误/健康度
  │  模块5 仅内存采集（不写入项目级/全局级），标记待下次恢复
  │  模块6 跳过（依赖写入能力）
  │  每次仍尝试写温区/冷区/索引 → 成功则计入恢复计数
  │
  ├── 触发条件: L0 规则文件 ~/.agents/AGENTS.md 不可读
  │     （仅影响模块3.3记忆健康度对比，模块3.1+3.2继续运行）
  │     但若同时满足 L2 条件 → 升级到此级（模块3全跳）
  │     OR 温区+冷区同时连续2次写入失败
  │     OR harvests/ 目录整体不可访问
  │
  ▼
L4 单行摘要（极限降级）
  │  输出: 📋 {项目} —— {1句总结} | ✅{N} ⏳{N} ⚠️ 降级运行
  │  模块5+6 完全跳过
  │
  └── 触发条件: harvests/ 整体不可访问 + L0 规则不可读
        OR 连续3次任意写入失败（L3时累计）
```

### 降级状态持久化

降级状态写入 `$HARVEST_BASE/_degradation.json`：

```json
{
  "level": "L1",
  "triggered_by": "温区 errors.md 写入失败",
  "fail_count": {"warm_write": 1, "cold_write": 0, "index_write": 0, "warm_read": 0, "cold_read": 0},
  "recovery_count": {"warm_write": 0, "cold_write": 0, "index_write": 0},
  "first_fail_at": "2026-06-28T19:30:00",
  "last_check_at": "2026-06-28T19:48:00",
  "consecutive_fails": 1,
  "history": [
    {"at": "2026-06-28T19:30:00", "event": "degrade", "from": "L0", "to": "L1", "reason": "温区 errors.md 写入失败"},
    {"at": "2026-06-28T19:48:00", "event": "check", "level": "L1", "attempt_write": "failed"}
  ]
}
```

**恢复规则（逐级恢复，非跳回L0）**：
- 降级后，每次 `/总结` 仍尝试写入（单次，不重试），用于检测恢复条件
- **L4→L3**: 连续2次温区写入成功
- **L3→L2**: 连续2次冷区写入成功 + L0 规则文件恢复可读
- **L2→L1**: 连续2次温区+冷区同时写入成功
- **L1→L0**: 连续2次温区+冷区+索引全部写入成功
- 恢复成功 → 清除对应 `fail_count` 和 `recovery_count`，更新 level
- `consecutive_fails` 计数器跨会话持久化（不随 session 重启重置）
- 完全恢复 L0 后 → 删除 `_degradation.json`

### 降级链输出格式

每次 `/总结` 输出末尾追加降级状态行：

```
⚠️ 降级 L1 (模块4仅内存) | 温区写入失败(1次) | 仍尝试写入 | 恢复需连续2次成功
⚠️ 降级 L2 (跳过记忆健康度+错误) | 连续2次温区写入失败 | 检查磁盘空间
⚠️ 降级 L3 (仅进度+事实) | 模块5仅内存采集(不写入) | L0规则不可读 | 检查 ~/.agents/AGENTS.md
⚠️ 降级 L4 (单行摘要) | 持久化层完全不可用 | 检查 ~/.agents/skills/summarize/harvests/
```

## 模块 3：建议总结 — 怎么优化

### 3.1 链路优化（基于台账模式匹配 + headroom 代理健康检查）

| 检测模式 | 触发条件 | 优化建议 |
|---------|:--------:|---------|
| 重复构建失败 | 连续2+会话构建失败 | "建议 `go mod tidy` 前置" |
| 环境缺失 | 缺工具/依赖/服务 | "创建环境检查脚本" |
| 重复错误尝试 | 重试≥2次未换方向 | "设2次失败后自动换路" |
| 配置遗漏 | 缺 .env/配置 | "创建 .env.example + 校验" |
| 重复失败模式 | 台账同类≥3次 | "提升为项目约定" |
| 无效步骤 | 做了无用操作 | "改前先 Read 确认" |
| Headroom 静默降级 | proxy.log 含 `rust_core_missing` / `python_only_degraded` | "检查 headroom Rust core: `pip install headroom-ai[ml]`" |
| Headroom 0 节省 | proxy_savings.json lifetime.tokens_saved=0 | "确认代理是否生效: 检查 HEADROOM_REQUIRE_RUST_CORE 环境变量" |

> **Headroom 健康检查**：每次运行时 Read `~/.headroom/logs/proxy.log` 末 20 行 + `~/.headroom/proxy_savings.json`，检测上述两种异常并将结果记入冷区 error-ledger.md + 温区 errors.md。

```
⚡ 链路优化({N}):
  ├ {模式}: {1句建议}
  └ 依据: {N}次重复 | {示例}
```

### 3.2 知识沉淀

## 模块 4：错误总结 — 出了什么问题

### 失败6类

| 代码 | 类型 | 检测方式 |
|:----:|------|---------|
| TOOL_ERR | 工具报错 | API error / denied / timeout |
| EXEC_FAIL | 执行失败 | build error / test fail |
| ENV_MISS | 环境缺失 | 缺命令/依赖/服务 |
| CFG_MISS | 配置缺失 | 缺 .env/配置文件 |
| RETRY | 重复尝试 | 重试≥2次未换方向 |
| LOGIC | 逻辑偏差 | 用户纠正/无用功 |

### 5维分类 ↔ 6类失败 映射

> 自进化闭环使用5维分类（PROC/ASSU/ENVR/TOOL/KNOW），模块4收割使用6类失败。以下为双向映射关系，确保温冷区数据与 L0 规则体系无缝对接。

```
5维分类（AGENT-MEMORY-SYSTEM.md）  →  6类失败（SKILL.md 模块4）
─────────────────────────────────────────────────────────
PROC  流程违规                     →  RETRY（重试未换路）
      未遵守现有规则/流程               LOGIC（跳过必须步骤/无用功）
─────────────────────────────────────────────────────────
ASSU  假设错误                     →  CFG_MISS（假设配置存在）
      基于错误假设行动                  LOGIC（编造API/路径、未确认前提）
─────────────────────────────────────────────────────────
ENVR  环境问题                     →  ENV_MISS（缺命令/依赖/服务）
      外部环境/工具限制                 EXEC_FAIL（网络被墙、权限不足）
─────────────────────────────────────────────────────────
TOOL  工具误用                     →  TOOL_ERR（选错工具/参数错误）
      选错工具或参数错误                LOGIC（old_string不匹配）
─────────────────────────────────────────────────────────
KNOW  知识盲区                     →  CFG_MISS（不知道项目配置约定）
      缺少领域/项目知识                 LOGIC（不了解项目结构）
                                  →  ENV_MISS（不知道所需工具链）
```

**使用规则**：
- 模块3 收割时使用 **5维分类**（写入 L0 热区 / 温区 / 冷区）
- 模块4 输出时使用 **6类失败**（屏幕显示）
- 温区 errors.md 条目同时记录两个分类字段：`5维: PROC | 6类: RETRY`
- 冷区 error-ledger.md 使用5维分类（与 L0 体系一致）

## 模块 5：成功经验收割 — 什么做对了 · 如何复用

> **定位**: 与模块4(错误轨)对称的**成功轨**。自动采集会话中的成功操作模式、工具效用对比、最佳执行路径，写入分层记忆供后续 Agent 复用。
> **设计来源**: CFGM 多粒度记忆提取 + MIRIX 程序记忆 + Mem0 向量图双引擎思想，简化为 Prompt 内规则驱动。

### 5.1 自动采集触发条件

在会话过程中，当以下场景发生时自动记录（内存暂存，/总结 时批量写入）：

| 触发场景 | 采集内容 | 记录类型 |
|---------|---------|:--:|
| 工具调用成功且非平凡操作 | 工具名 + 参数模式 + 耗时 + 结果摘要 | `tool_success` |
| 同一任务使用多种工具并对比 | 工具A→结果 vs 工具B→结果 + 效率对比 | `tool_compare` |
| 搜索→定位→修改 完整成功链路 | 完整执行路径 + 跳过的错误分支 + 总耗时 | `best_path` |
| 错误后成功恢复 | 错误类型 → 修复策略 → 成功结果 | `recovery_pattern` |
| 用户明确表扬/确认/采纳 | 当前操作模式 + 用户反馈标记 | `user_approved` |

**非平凡判定**：以下操作不触发采集（过于常规）：
- 单次 read_file 且无后续修改
- 单次 list_dir 浏览
- 单次 search_content 且未产生决策
- 任何被用户纠正后放弃的操作

### 5.2 工具效用对比规则

当同一会话中出现「同一意图、不同工具」的调用时，/总结 时自动生成对比记录：

**对比触发条件**：
- 同一任务目标（如"查找函数定义"、"搜索文件"、"获取网页内容"）
- 使用了 ≥2 种不同工具
- 结果存在可量化的效率差异（耗时/精度/结果数量）

**对比输出格式**：
```
📊 工具效用对比({N}):
  ├ 任务: {1句任务描述}
  │  路径A: {工具A} → {结果A} → 耗时 {t1}s
  │  路径B: {工具B} → {结果B} → 耗时 {t2}s
  │  结论: {工具B} 效率高 {倍数}x | 建议: {1句使用建议}
  └ → 写入: 全局级 (工具选择策略)
```

**对比维度**：
| 维度 | 比较内容 | 示例 |
|------|---------|------|
| 速度 | 实际耗时对比 | "3.2s vs 0.8s" |
| 精度 | 结果数量/相关性 | "47个(含噪声) vs 3个(全精准)" |
| 可靠性 | 是否稳定复现 | "2次均成功 vs 1次超时" |
| 成本 | token 消耗估算 | "~2000 token vs ~500 token" |

### 5.3 最佳执行路径提取

基于 CFGM 粗→细粒度思想简化：

```
Step 1: 识别任务类型 — 通过操作特征匹配已知类型或创建新类型
Step 2: 记录本次路径 — 工具序列 + 分支决策 + 结果状态 + 耗时
Step 3: 对比历史同类 — 查找此前记录的同类任务最佳路径
Step 4: 更新最优路径 — 若本次更优(更快/更准/更稳) → 更新；若首次 → 创建
```

**路径模板**（写入项目级或全局级记忆）：
```json
{
  "task_type": "symbol_search_and_modify",
  "best_path": ["codegraph explore", "read_file", "replace_in_file"],
  "avoid_path": ["grep 全项目搜索", "逐个文件 read_file 遍历"],
  "efficiency_gain": "3-5x",
  "last_verified": "2026-06-28",
  "success_count": 12,
  "fail_count": 0
}
```

**任务类型识别表**（持续扩展）：
| 任务类型 | 特征 | 推荐路径 |
|---------|------|---------|
| `symbol_search` | 查找函数/类/变量定义或引用 | codegraph → grep(降级) |
| `file_explore` | 了解项目结构/目录 | codegraph explore → list_dir |
| `text_search` | 搜索日志/注释/配置值 | grep/search_content |
| `code_modify` | 读→改→写文件 | read_file → replace_in_file |
| `web_research` | 获取外部信息/文档 | web_search → web_fetch(深度) |
| `error_diagnose` | 定位+修复错误 | read_lints → read_file → replace_in_file |
| `multi_file_refactor` | 跨文件重构 | codegraph impact → 批量 replace_in_file |

### 5.4 成功经验分类标签

继承模块3.2的4类标签，新增2个维度：

| 标签 | 含义 | 示例 | 默认写入层级 |
|------|------|------|:--:|
| 🏗️ 架构 | 项目结构/模式发现 | "此项目用 monorepo，修改需同时更新3个包" | 项目级 |
| ⚠️ 坑点 | 已避开的陷阱 | "PowerShell 不支持 &&，用 ; 和 if ($?) 替代" | 全局级 |
| 🔧 配置 | 环境/工具配置经验 | "此项目 Python 3.12，需用 uv 而非 pip" | 项目级 |
| 📐 约定 | 项目代码规范 | "错误处理统一用 result pattern" | 项目级 |
| ⚡ 路径 | **新增**: 最佳执行路径 | "符号查找优先 codegraph，降级用 grep" | 全局级 |
| 🔬 效用 | **新增**: 工具效用对比 | "WebFetch vs web_search: 前者适合长文，后者适合快速搜索" | 全局级 |

### 5.5 输出格式

```
⚡ 成功经验({N}):
  ├ ⚡ 路径: 符号查找→优先 codegraph(0.8s)，降级 grep(3.2s)
  ├ 🔬 效用: WebFetch 长文提取 > web_search，后者适合快速搜索

  │  路径B: codegraph callers → 3个结果(全精准) → 0.8s
  │  结论: codegraph 效率高 4x，精度高 15x
  └ → 写入: 全局级 (工具选择策略)
```

## 模块 6：分类路由与分层写入 — 写到哪

> **定位**: 将模块4(错误)和模块5(成功)采集的经验，按分类路由写入对应层级文件。拿不准的进入待确认队列询问用户。

### 6.1 写入层级决策树

```
经验记录
  │
  ├── 涉及具体项目路径/文件/配置/架构？
  │     └── YES → 项目级 (./.agents/memories/ 或项目记忆文件 $PROJ_MEMORY)
  │
  ├── 涉及通用工具选择策略/Agent行为模式/跨项目通用？
  │     └── YES → 全局级 ($L0_MEM_DIR 或 $L0_RULES)
  │
  ├── 涉及具体用户偏好/个人约定？
  │     └── YES → 全局级 ($L0_MEM_DIR/user-preferences.md)
  │
  ├── 涉及特定技能执行优化？
  │     └── YES → 技能级 ($SKILL_BASE/summarize/references/learned/)
  │
  └── 无法确定归属（同时满足多个层级 OR 无明确归属）？
        └── → 待确认队列 (_pending.json) → /总结 时询问用户
```

### 6.2 分层写入目标文件

> **路径使用平台变量**，实际值由「平台适配」节检测结果决定。下表列出变量名，各平台自动替换。

| 层级 | 路径（变量） | 内容 | 写入触发 |
|------|------|------|---------|
| L0 全局规则 | `$L0_RULES` | 通用行为规则、工具选择策略 | 效用对比结论、通用最佳路径 |
| L0 全局记忆 | `$L0_MEM_DIR` | 用户偏好、全局经验 | 用户偏好发现、跨项目通用经验 |
| 项目级记忆 | `./.agents/memories/` | 项目结构、配置、约定 | 项目特定经验、架构发现 |
| 项目摘要 | `./$PROJ_MEMORY` | 项目记忆摘要（供 agent 读取） | 每次/总结 时更新 |
| 技能级 | `$SKILL_BASE/summarize/references/learned/` | 技能执行经验 | 技能特定优化 |
| 待确认 | `$HARVEST_BASE/_pending.json` | 无法归类的经验 | 不确定时暂存 |

### 6.3 待确认队列机制

**写入规则**：
- 经验同时满足多个层级归属 → 进入待确认
- 经验与已有规则可能冲突 → 进入待确认（标 `[冲突]`）
- 经验涉及用户主观偏好且无历史记录 → 进入待确认

**存储格式**（`$HARVEST_BASE/_pending.json`）：
```json
{
  "pending": [
    {
      "id": "pend_001",
      "content": "PowerShell 脚本中避免使用 curl，用 Invoke-WebRequest",
      "candidates": ["全局级(跨项目通用)", "项目级(仅本项目用PS)"],
      "context": "在 scripts/ 目录下编写 .ps1 时发现",
      "created_at": "2026-06-28T20:00:00",
      "status": "pending"
    }
  ]
}
```

**交互方式**：在 /总结 输出末尾，模块6 区域显示：
```
❓ 待确认({N}):
  [1] "PowerShell 脚本避免 curl，用 Invoke-WebRequest"
      候选: A)全局级 B)项目级 C)跳过
  [2] "此项目 Go 版本锁定 1.22" [冲突: 项目 go.mod 写 1.21]
      候选: A)以代码为准 B)以经验为准 C)保留两者
  回复 "确认 1A 2A" 完成分类
```

**确认后处理**：
- 用户回复 → 写入对应层级，从 _pending.json 移除
- 连续3次/总结 未确认 → 自动降级写入项目级（保守策略），标 `[auto]`
- 用户选择"跳过" → 从 _pending.json 删除

### 6.4 输出格式

```
📝 写入({N}):
  ├ → 项目级(2): 🏗️架构: monorepo结构 | 🔧配置: Python 3.12+uv
  ├ → 全局级(1): ⚡路径: codegraph优先策略
```

## Headroom 集成（v6.4.0+）

> 每次 `总结` 自动检查 headroom 代理健康与节省统计。

### 检测链路

```
Read ~/.headroom/logs/proxy.log (末20行)
  → ERROR/WARNING 含 rust_core_missing/python_only_degraded
  → 记入 error-ledger.md (PROC | TOOL:Headroom SIG:rust_core_missing)
  → 模块3 输出 "🪖 Headroom: 降级 ⚠️ (0 token)"

Read ~/.headroom/proxy_savings.json
  → lifetime.tokens_saved / total_input_tokens / compression_savings_usd
  → 记入 _self-stats.md Headroom 统计节
  → 模块3.3 输出 "🪖 Headroom: 正常 ({N} token) ｜ {引擎}"
```

### 输出格式

- 降级状态 → 记忆健康度行 `🪖 Headroom: 降级 ⚠️` → 高风险预警
- 正常状态 → `🪖 Headroom: 正常 ({节省} token) ｜ Kompress/基础`
- 0 节省 → `🪖 Headroom: 待确认 (0 token)` → 建议检查环境变量

### 错误记录

| 错误类型 | 分类 | 涉实体 | 避免规则 |
|---------|:--:|--------|---------|
| Headroom Rust core 缺失 | PROC | TOOL:Headroom SIG:rust_core_missing | 执行 `pip install headroom-ai[ml]` |
| Headroom 代理未生效 (0 savings) | ENVR | TOOL:Headroom FILE:proxy_savings.json | 检查 HEADROOM_REQUIRE_RUST_CORE |

### 双轨存档（v6.4.0+）

`总结 压缩` 执行时新增可选 headroom 加速步骤：

```
1. 明文压缩: archive/{date}-errors-compact.md  (与原行为一致)
2. headroom_compress(明文内容) → hash
3. hash 写入 archive/{date}-errors-compact.md.hr (纯文本文件，内容 = hash)
4. <可选> index.md 每行可追加 headroom_hash 列

检索优先级:
  ① 有 headroom MCP → headroom_retrieve(hash) → 解压 (更快)
  ② 无 headroom     → 读取明文 .md 文件 (fallback)
```

**降级安全**：headroom 不可用时，`总结 压缩` 退化为纯明文模式，行为与 v6.3 完全一致。

### `总结 深度 写入` — 自编辑规则（自进化闭环关键路径）

> **触发条件**: 模块4 收割判定某条温区错误满足「14d内复发≥2次」且「热区<5条有空位」。

#### 完整操作步骤

```
Step 1: Read — 加载 L0 规则文件
  → Read ~/.agents/AGENTS.md
  → 定位 P0 热区表（搜索 "| 错误类型 | 次数 | 分类 | 避免规则 | 状态 |"）
  → 解析当前热区条目数、最旧条目、clean天数

Step 2: 定位 — 计算插入位置
  → 若热区<5条 → 在 P0 表末行追加新条目
  → 若热区=5条 → 先执行定容（Step 3）再插入
  → 新条目格式:
    | {错误简述} | {复发次数} | {5维分类} | {1句避免规则} | 🆕 |
  → 若条目已存在（相同错误简述）→ 更新次数+状态为 🔄复发，不新增行

Step 3: 定容 — 热区≤5条约束
  → 若当前热区=5条且需插入新条目:
    a. 找最旧1条（按 clean天数 或 首次记录时间 排序）
    b. 将该条目从 P0 表移除
    c. 降级写入温区 errors.md（追加到项目 errors.md 头部，标 [L0降级]）
    d. 在 P0 表插入新条目
  → 输出: "📏 定容: {旧规则} → 温区 | {新规则} → 热区"

Step 4: 插入 — Edit 写入
  → 使用 Edit 工具精确替换 P0 表区域
  → old_str 包含完整 P0 表（确保唯一匹配）
  → new_str 为更新后的完整 P0 表
  → 若 Edit 失败（old_str 不匹配）→ 重新 Read AGENTS.md → 重试1次
  → 若仍失败 → 输出 "⚠️ L0 回写失败: {原因}" → 降级写入温区，标 [L0回写失败]

Step 5: 回测 — 验证写入结果
  → Read ~/.agents/AGENTS.md → 确认 P0 表已更新
  → 检查: 条目数≤5、格式正确、无重复
  → 输出: "✅ 回测通过: P0表 {N}/5"

Step 6: commit — 版本记录
  → git add ~/.agents/AGENTS.md
  → git commit -m "summarize: 升级温区规则到热区 [{错误简述}]"
  → 若 git 不可用 → 跳过，输出 "⚠️ Git 不可用，规则已写入但未提交"

Step 7: 更新温区 — 标记来源
  → 在温区 errors.md 对应条目追加 "→ 🔥 已升级热区 {date}"
  → 温区条目不清除（保留升级历史）
```

```
  ├ Read: ~/.agents/AGENTS.md (P0表 {N}/5)
  ├ 定位: 热区有空位，追加到第{N+1}行
  ├ 定容: 无需（当前{N}条）
  ├ 插入: "P0-{规则简述}" → P0表
  ├ 回测: ✅ P0表 {N+1}/5
```

#### 错误处理

| 场景 | 行为 |
|------|------|

## 格式原则

- **六柱清晰**：项目·进度·建议·错误·成功经验·分层写入，不混入无关内容
- **一行一事**：不堆砌无关信息
- **上限折叠**：超限自动折叠，不突破25行
- **增量必控**：每次输出检查所有层级是否超限
- **超限必缩**：超限时自动触发压缩建议
- **常驻可见**：热区+记忆标注 KB 估测
- **建议非待办**：用户自主选择是否采纳
- **成功必记**：非平凡成功操作自动记录，工具效用必对比
- **写入必显**：分层写入操作日志在输出末尾全显，待确认项不可折叠
