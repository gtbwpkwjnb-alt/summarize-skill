# /总结 v3.0 — 进度·错误·收割·进化

**四重闭环：进度快照 → 错误诊断(含回测) → 经验收割 → 优化建议。**
**错误分三级存储：全局 / 项目 / 代码。技能自身持续进化。**

---

## 主动建议规则

> 即使未收到 `/总结` 命令，在以下条件触发时主动提醒用户：

| 条件 | 阈值 | 提醒语 |
|------|:--:|------|
| 会话轮次过多 | ≥ 20 轮 | "💡 已 {N} 轮，建议 /总结 压缩上下文" |
| 错误频繁 | ≥ 3 个 | "⚠️ 本会话已 {N} 个错误，建议 /总结 收割经验" |
| 工具调用密集 | ≥ 30 次 | "🔧 工具调用 {N} 次，建议 /总结 归档进度" |

**提醒时机**: 回复末尾追加一行，不打断当前任务。同一会话每种条件只提醒一次。

## 核心架构：三级错误存储

```
harvests/
  error-ledger.md              ← 全局错误（跨项目复用，如"Edit前未Read"）
  {project}/
    errors.md                  ← 项目错误（仅本项目，如"CC-Switch不兼容"）
    YYYY-MM-DD-{id}.md         ← 会话收割文件
  _self-stats.md               ← 技能自身运行统计
  index.md                     ← 收割索引
```

**分流规则**：
- 错误涉及"工具使用方式/通用流程" → 全局 `error-ledger.md`
- 错误涉及"特定项目配置/架构/领域" → 项目 `{project}/errors.md`
- 错误涉及"代码实现细节" → 不入账本，转为代码优化建议

### 模块 2：⚠️ 错误诊断（增强：分级 + 回测 + 预防）

**核心增强：错误分全局/项目存储，每次 /总结 时执行回测验证。**

#### 2.1 本次错误检测

**自动检测规则**（无需显式标记）：
- 工具调用返回 error/失败
- 用户指出错误或纠正 AI
- 同一操作重试 3+ 次
- Edit 失败（old_string 不匹配）

**根因分类**（5 维度）：

| 分类 | 代码 | 定义 | 示例 |
|------|:--:|------|------|
| 流程违规 | PROC | 未遵守现有规则/流程 | Edit前未Read、未验证就说完成 |
| 假设错误 | ASSU | 基于错误假设行动 | 编造API名称、假设文件路径存在 |
| 环境问题 | ENVR | 外部环境/工具限制 | GitHub被墙、cmd.exe不支持Unix命令 |
| 工具误用 | TOOL | 选错工具或参数错误 | 用Bash当Grep、Edit的old_string不匹配 |
| 知识盲区 | KNOW | 缺少领域/项目知识 | 不知道某个配置项、不了解项目结构 |

#### 2.2 错误表

```
| 错误 | 原因 | 分类 | 作用域 | 历史次数 | 标记 |
|------|------|------|:-----:|:------:|:--:|
| {简述} | {根因} | {PROC/ASSU/ENVR/TOOL/KNOW} | 全局/项目 | {N}次 | 🆕/⚠️/⚠️⚠️⚠️ |
```

**标记规则**：首次→🆕 | 2次→⚠️ | 3+次→⚠️⚠️⚠️ 需沉淀规则

#### 2.3 🛡️ 回测验证（新增）

对比本次会话行为与已沉淀规则，输出遵守报告：

```
🛡️ 规则遵守报告:
✅ "{规则}" — 连续 {N} 次操作均遵守，干净天数 +1
⚠️ "{规则}" — 本次会话违反！干净天数归零
🔄 "{规则}" — 安静 {N} 天后复发，需检查规则是否不够具体
```

**回测逻辑**：
1. 读取 `RULES.md` 中所有 P0/P1 规则
2. 扫描本次会话，检查每条规则是否被遵守
3. 更新 `error-ledger.md` 中各错误的 `days_clean` 字段
4. 复发时标注 🔄 并建议修订规则

#### 2.4 高频警告

```
⚠️⚠️⚠️ 高频错误: "{错误类型}" — {N} 次（跨 {M} 项目）
建议: {具体行动}
```

### 模块 3：🚜 经验收割

**合并旧版模块 6+7+8，一次完成标注→进化→写入。**

#### 3.1 自动标注

扫描全对话，提取四类标注。无显式标记的自动推断：

```
[DECISION] 决策记录
- {决策内容} | 原因: {why} | 备选: {alternatives}

[ERROR] 错误记录
- {错误描述} | 根因: {root cause} | 修复: {fix} | 预防: {prevention}

[INSIGHT] 洞察记录
- {发现} | 领域: {domain} | 建议: {actionable advice}

[RULE] 规则建议
- {建议规则} | 触发场景: {when} | 优先级: P0/P1/P2 | 作用域: 全局/项目/代码
```

**自动推断规则**：
- 用户说了"保留XX"/"不要YY" → `[DECISION]`
- 工具失败 + 修复成功 → `[ERROR]`
- 跨领域发现 → `[INSIGHT]`
- 新增错误模式/规则 → `[RULE]`（自动判断作用域）

#### 3.2 规则进化（三级分流）

```
新增规则建议（按作用域分流）:

[全局] → AGENTS.md
- {规则} | 触发: {when} | 优先级: {P} | 来源: {会话}

[项目] → {project}/.zcode/AGENTS.md
- {规则} | 触发: {when} | 优先级: {P} | 来源: {会话}

[代码] → 优化建议（见模块4）
- {建议} | 来源: {会话}

高频错误警告:
⚠️⚠️⚠️ "{错误类型}" 已 {N} 次（跨 {M} 会话），建议加入对应层级 AGENTS.md
```

#### 3.3 收割写入

**Step 1: 确定项目名** — 从当前工作目录提取。

**Step 2: 写入收割文件**

路径：`harvests/{project}/YYYY-MM-DD-{id}.md`
格式：6 分区 — `activeContext` `progress` `decisions` `errors` `patterns` `nextSteps`
详见 `references/harvest-format.md`

**Step 3: 更新错误账本**

- 全局错误 → `harvests/error-ledger.md`
- 项目错误 → `harvests/{project}/errors.md`（不存在则创建）
- 更新 `days_clean`、`last_seen`、`verified` 字段

**Step 4: 更新索引** — 追加到 `harvests/index.md`

**Step 5: 确认输出**

```
✅ 已收割 — {project} 项目
   文件: harvests/{project}/YYYY-MM-DD-{id}.md
   全局错误: {N} 条（新增 {M}，高频 ⚠️⚠️⚠️ {H}）
   项目错误: {N} 条（新增 {M}，高频 ⚠️⚠️⚠️ {H}）
   标注: {D} 决策 / {E} 错误 / {I} 洞察 / {R} 规则建议
   回测: {通过} 条规则遵守 / {违反} 条规则违反 / {复发} 条错误复发
```

**Step 6: 更新自反馈统计** — 追加到 `harvests/_self-stats.md`

## 子命令

### `/总结 历史`

1. 读取 `harvests/index.md`
2. 展示按项目分组的收割列表
3. 等待用户输入编号查看详情
4. 读取对应收割文件并展示摘要

### `/总结 错误`

1. 读取 `harvests/error-ledger.md`（全局）+ `harvests/{project}/errors.md`（本项目）
2. 分两区展示：全局错误 | 本项目错误
3. 按频率降序排列，3+ 次高亮
4. 展示回测状态（days_clean、复发标记）
5. 建议沉淀为规则的条目

### `/总结 统计`

1. 读取 `harvests/_self-stats.md`
2. 展示累计统计：总运行次数、检测错误、规则建议、平均压缩率
3. 最近 5 次运行记录
4. 技能自进化建议 — 哪些模块产出低、哪些维度需增强、建议裁剪什么
5. **🆕 版本检查** — 读取本地 `VERSION` 文件，调用 GitHub API (`https://api.github.com/repos/gtbwpkwjnb-alt/summarize-skill/releases/latest`) 比对，如有新版本提醒升级：

```
📦 版本检查:
   当前: v3.0.0 | 最新: v3.1.0 🆕
   更新: cd ~/.agent-skills/summarize && git pull
   变更: {changelog摘要}
```

## 格式原则

- **先结论后过程** — 每个模块先说结果
- **一行一事** — 不堆砌
- **高频高亮** — 3+ 次错误用 ⚠️⚠️⚠️
- **收割内聚** — 所有持久化数据在 `harvests/` 内，不污染项目目录
- **待办→建议** — 所有产出物用"建议"而非"待办"，由用户自主选择

---

## 错误记录字段规范

每条错误在账本中的标准格式：

```markdown
| 错误类型 | 次数 | 分类 | 作用域 | 首次 | 最近 | days_clean | 避免规则 | 已验证 | 涉及项目 |
|---------|:---:|:--:|:-----:|------|------|:--------:|---------|:-----:|---------|
| {类型} | {N} | {CODE} | 全局/项目 | {date} | {date} | {N} | {规则ID} | ✅/❌ | {项目} |
```

**新增字段说明**：
- `days_clean`: 自上次出现后连续未再犯的天数（正向激励）
- `避免规则`: 对应的预防规则 ID，如 `P0-Edit前Read`
- `已验证`: 该规则是否经回测确认有效
