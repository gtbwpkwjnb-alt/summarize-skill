# 总结 v5.9.0 — 🛡️ Error Immune System: 错误免疫系统 · 精炼 · 进度 · 自进化 · 可逆 · 跨Agent（多平台通用）

> **定位**: 🛡️ AI 编程助手的错误免疫系统 — 自动收割错误→多维分类→自进化规则→收敛预警。
> **设计目标**: 任务开发>1天时，让你扫一眼就了解会话全貌。

## 归档模型

> 每个会话一个归档文件，每次 `总结` **覆盖更新**（非追加）。首次运行自动创建 `harvests/` + `error-ledger.md` + `_self-stats.md`。

### 三级存储层级

| 层级 | 文件 | 大小上限 | 加载策略 |
|:--:|------|:--:|------|
| 🔥**热** | `AGENTS.md <error_patterns>` 表 | ≤5条 | 每次会话硬加载 |
| 🔶**温** | `harvests/{project}/errors.md` | ≤30条 | 语义检索按需加载 |
| 🔵**冷** | `harvests/error-ledger.md` | ≤100条 | 总结时+主动检索 |
| ⬜**归档** | `harvests/archive/{date}-rules.md` | 无上限 | 显式恢复 |

```
harvests/
  error-ledger.md              ← 冷区：全局错误（跨项目复用，≤100条）
  _self-stats.md               ← 技能自身运行统计
  index.md                     ← 收割索引（同会话只保留最新一条）
  archive/                     ← 超限规则归档
    {date}-{N}rules.md
  {project}/
    errors.md                  ← 温区：项目错误（≤30条）
    {session-id}.md            ← 当前会话归档（覆盖更新）
```

**归档文件内容** (`harvests/{project}/{session-id}.md`)：
```markdown
# {项目名} — {session-id} 归档
> 最后更新: {时间戳}

## 压缩上下文（跨会话交接）
**目标**: {1句任务目标}
**当前状态**: {1句当前进度}
**关键决策**: {决策列表，每条附带原文锚定引用}
**使用技能**: {本次会话调用的技能及次数}
**产出物**: {文件/代码变更清单}

## 完整文件清单
{所有涉文件，不截断}

## 完整错误列表
> 每条错误附带原文锚定引用，防止反向理解偏差。

| 错误 | 分类 | 原文引用 |
|------|:--:|---------|
{完整错误列表，不截断，每条附带原文锚定}

## 技能调用详情
{每次技能调用的时间、token估算、产出}
```

**更新策略**：
- 每次 `总结` → 覆盖写入归档文件（文件头更新 `最后更新` 时间戳）
- `index.md` 同会话只保留最新一条索引记录
- 旧版 `{date}-{session}.md` 格式文件在下次总结时自动迁移到 `{session-id}.md`
- **跨Agent**: 下一会话 Agent 读取 `index.md`→匹配 `{project}`→加载 `## 压缩上下文` 节；无索引则跳过
- **Git版本化**: 每次 `总结` 自动 commit `harvests/` + `AGENTS.md` 变更（规则/归档变更可 rollback 到任意历史状态）
- **块超限处理**: 某层超限时，clean周期最长的规则降级（热→温→冷→归档），输出 `↕️ 升降级日志`

## 模块 3：错误自进化（核心）

> 执行前加载 AGENTS.md 的 `<error_patterns>` 表作为规则源，错误历史统计见 `harvests/error-ledger.md`。规则 ID 必须与 AGENTS.md 一致。
> ⚠️ 降级: 项目无 AGENTS.md 或缺少 `<error_patterns>` 表时，模块3仅做错误统计，跳过规则回测和建议。

**错误分流**: 工具/流程错误→全局`error-ledger.md` | 项目配置→`harvests/{project}/errors.md` | 代码实现→模块3建议不入账

**检测**:
- **工具报错**: API error / denied / not found / timeout → 可收割
- **用户纠正**: 用户指出输出错误/方向偏差 → 可收割
- **重试≥2次**: 同一操作连续2次失败 → 无论原因，必须收割（对齐 P0-换路）
- **注意**: "用户纠正"与"工具报错"分开——用户说"不对，改成X"是意图偏差（KNOW/ASSU），工具返回403是环境问题（ENVR）

**采纳验证**: 扫描当前会话的工具调用序列 → 与 AGENTS.md 热区规则比对:

  - **已遵守**: Edit 前有 Read → `P0-改前先读` ✅
  - **已触发但违背**: 重试≥2次未换方向 → `P0-换路` ❌
  - **未触发**: 本次无对应场景 → 不判断，标记 `—`
  - 输出 `📋 规则采纳追踪` 表（见下模板）

**三级检索（多信号融合）**: 每次总结前，按优先级检索三级错误存储。采用三重评分融合确定相似度：
  `score = semantic*0.4 + keyword*0.3 + temporal*0.3`
  - **语义匹配**: 错误描述语义相似度
  - **关键词匹配**: 当前错误类型 vs 存储错误类型的关键词重叠率
  - **时间接近度**: `temporal_score = max(0, 1 - days_since/30)`，近期错误权重更高
  - **频率加权**: `freq_boost = log(1 + count)`，高频错误信号自然增强
  - 三区统一使用此评分，温/冷区阈值分别设为 >0.5 / >0.3
- 🔥热区（AGENTS.md）：已加载 → 检查本次是否触犯已有规则
- 🔶温区（`{project}/errors.md`）：多信号评分 >0.5 → 匹配则累计不新增
- 🔵冷区（`error-ledger.md`）：多信号评分 >0.3 → 按 clean 天数 + freq_boost 判定是否回温
- 温/冷区检索结果写入 `↕️ 升降级` 输出段

**冲突检测**: 新规则写入前比对热区已有规则——若描述重叠/矛盾则输出冲突标注，建议合并而非新增。

**实体链接**: 错误描述自动关联实体类型（写入 error-ledger.md `涉实体` 列）:
  - **工具类**: 提取工具名 (Read/Write/Bash/Grep/Glob/Edit) → 标记 `TOOL:{工具名}`
  - **文件类**: 提取文件后缀 (*.ts/*.py/*.md/*.json/*.go) → 标记 `FILE:{后缀}`
  - **项目类**: 提取项目名 (ZCode/CowAgent/mem0等) → 标记 `PROJ:{项目}`
  - **环境类**: 提取错误信号 (timeout/denied/not found/403/500) → 标记 `SIG:{错误码}`
  - 实体关联用于:\
    ① 多信号检索关键词匹配加权\
    ② 因果链分析起点判定\
    ③ 温区按文件/项目分组统计

**因果链分析**: 按 5 维分类建立错误传播路径，识别"A→B→C"链条模式:

  ```
  PROC(跳过步骤) → TOOL(误用工具) → ENVR(环境报错)
  ASSU(假设错误) → KNOW(知识盲区)
  PROC(不改思路) → TOOL(命令参数错) → ENVR(timeout/denied)
  ```
  检测方法: 扫描同会话连续错误 → 按时间顺序排列 → 匹配传播模式
  - 匹配成功 → 输出 `🔗 因果链: {前序} → {后续}`
  - 匹配到3+次同链条 → 标记 ⚠️⚠️⚠️ 并建议阻断第一环

**5 维分类**（通用版，按你的平台替换例子）：

| 代码 | 分类 | 说明 | 典型规则ID |
|:----:|------|------|-----------|
| PROC | 流程违规 | 跳过必要步骤 | P0-改前先读, P0-换路, P0-验证后声明 |
| ASSU | 假设错误 | 未确认的标识符/路径 | P0-验证后引用 |
| ENVR | 环境问题 | 网络/系统/权限限制 | P0-SSH优先, P1-harness状态 |
| TOOL | 工具误用 | 选错工具或参数 | P1-PowerShell变量, P1-关键读取降级 |
| KNOW | 知识盲区 | 缺乏领域/项目上下文 | P0-会话节流, P0-长会话压缩 |

```
⚠️ 高频错误 (≥2次, 按次数降序):
| {前5条高频错误} | {分类} | {N}次 | {趋势} | {标记}
  …等{N}条高频错误 📦（超5条时折叠）

🆕 新错误 (首次出现):
| {前3条新错误} | {分类} | {1次} |
  …等{N}条新错误 📦（超3条时折叠）

📋 规则采纳追踪:
| 规则 | 本次行为 | 状态 |
|------|---------|:--:|
| {P0-改前先读} | 本次Edit前均Read | ✅ 已遵守 |
| {P0-换路} | 本次无重试场景 | — 未触发 |
| {P0-验证后引用} | 本次Grep后引用 | ✅ 已遵守 |

🛡️ 规则有效性:
| 规则 | clean天 | 历史/月 | 当前/月 | 置信度 | 权重 | 效果 | 动作 |
|------|:--:|:--:|:--:|:----:|:--:|:--:|:--:|
| {P0-换路} | 4 | 3次 | 1次 | ✅ 高 | 1.0 | ✅有效 | 保持热区 |
| {P0-SSH优先} | 32 | 2次 | 0次 | 📶 中 | 0.5 | ➡️冗余 | →降温温区 |
| {P0-改前先读} | 0 | 2次 | 3次 | ⚠️ 低 | 1.0 | ❌复发 | 需修订 |
  > **置信度**: ✅高(≥3次+已验证+clean增长) / 📶中(2次+已验证) / ⚠️低(1次或未验证)
  > **衰减公式**: `weight = min(hist_count/3, 1) * 0.5^(clean_days/30)`
  > 规则权重 = (历史频率归一化) × (30天半衰期衰减)。权重<0.3 时规则自动标记 ➡️ 降级候选

↕️ 自动升降级:
  {P0-SSH优先} → 温区 (clean≥30d)
  {P0-回写闭环} → 候选热区 (14d内新违反×2)

📈 错误趋势 (较上月):
  统计方法: 读取 error-ledger.md → 按分类汇总上月 vs 本月错误数 → 计算变化率
| 分类 | 上月 | 本月 | 趋势 |
|------|:--:|:--:|:--:|
| PROC | 2次 | 3次 | 📈 +50% |
| ASSU | 2次 | 0次 | 📉 清零 |
| ENVR | 1次 | 2次 | 📈 +100% |
| TOOL | 2次 | 1次 | 📉 -50% |
| KNOW | 0次 | 1次 | 🆕 新增 |

⚠️ 收敛预警:
  检测方法: 扫描本周(7d) vs 上周同分类错误数 → 涨幅>50%且绝对数≥2 则预警
| 分类 | 上周 | 本周 | 涨幅 | 预警 |
|------|:--:|:--:|:---:|:---:|
| ENVR | 1次 | 3次 | +200% | 🟡 注意 |
| PROC | 2次 | 2次 | 0% | ✅ 稳定 |
  > 连续 2 周同一分类预警 → 标记 🔴 并建议增加一条热区规则

🛡️ 对抗性自检（每条规则建议执行前）:
  1️⃣ 反例检查: 当前会话中是否已有同类场景但未出错？→ 有则降低置信度
  2️⃣ 重叠检查: 是否与热区已有规则描述重叠？→ 有则建议合并而非新增（冲突检测已做）
  3️⃣ 泛化检查: 规则是否过于宽泛（如"小心所有工具"）？→ 要求附加具体触发条件
  输出: `✅ 自检通过` / `⚠️ 有疑虑（原因）: {规则ID} 因 {原因} 建议暂缓`

🌐 出站验证（首次出现的错误类型建议规则前执行）:
  - 提取错误类型关键信号（如 "timeout" / "denied" / "not found"）
  - 调用 WebSearch 或 agent-reach 搜索 1 条同类处理方案
  - 搜索结果匹配 → 标注 `🔗 参考: {来源}`
  - 搜索无明确方案 → 标注 `⚠️ 经验规则，待验证`
  - 目的: 减少基于单次事件生成幻觉规则

⚡ 进化:
全局→ {规则建议}
项目→ {规则建议}

✏️ 自编辑（一键追加至 AGENTS.md — 需用户授权）:

  2. 插入: 用 Edit 工具在表末尾追加新行:
     | `{分类}` | `{规则ID}` | `{说明}` | `{已验证}` | `{clean天}` |
  3. 定容: 追加后热区超 5 条 → 按 clean 天数升序将多余规则降级至温区

  4. 提交: `cd <project> && git add AGENTS.md && git commit -m "规则追加: {规则ID}"`

  > 直接替换已有行（当规则需修订时）:
  > 用 Edit 工具将 `{规则ID}` 所在行的说明字段替换为新内容
```
> 冲突检测: 新规则 {X} 与已有 {Y} 部分重叠，建议合并。

**标记**: 🆕首次 / ⚠️2次 / ⚠️⚠️⚠️≥3次

## 模块 4：技能调用统计

> 统计当前项目中用户级技能的调用次数和效果。仅统计技能（summarize/skills-audit 等），不含底层工具（Read/Write 等）。

**Token 数据来源**（优先级）：
1. 从 AI 助手的上下文窗口统计中提取使用率（输入框附近上下文统计数据）
2. 无数据时用字符估算：中文 ~1.5 字符/token，英文 ~4 字符/token
3. 估算值标记 `~`

```
📊 技能调用:
| {技能名} | {N}次 | token {~估算或-} | {效果}
```
> token 列：有数据时 `~{估算}`，无数据时 `-`。

**效果标记**：
| 标记 | 含义 | 示例 |
|:--:|------|------|
| ✅ | 节省 token | summarize 压缩上下文，净省 token |
| ⚠️ | 消耗 token | skills-audit 完整审计，净耗 token |
| ➡️ | 中性 | 工具性调用，无明显 token 影响 |

**详细数据**（每次调用的时间、token估算、产出）写入归档文件 `技能调用详情` 节。

## `_self-stats.md` 字段规范

> `harvests/_self-stats.md` 由每次 `总结` 自动更新覆盖。用于跨会话趋势追踪和自我健康度检查。

### 新增字段（v5.8.0）

| 字段 | 数据来源 | 说明 |
|------|---------|------|
| `月度错误趋势` | error-ledger.md 日期 | 按月汇总各分类错误数，计入 `📈 错误趋势` 列 |
| `规则效果月度变化` | 对比上月/本月 rule_validity | 有效规则数/无效规则数/复发规则数月度对比 |
| `分类漂移 (Δ%)` | 本月 vs 上月各分类占比 | 各分类错误占比变化，>10% 标记 🟡 关注 |
| `平均压缩率` | 归档模块1行数/总会话行数 | 低于 80% 标记 🟡 关注 |

### 示例格式

```
## 月度错误趋势
| 月份 | PROC | ASSU | ENVR | TOOL | KNOW | 总计 | Δ上月 |
|------|:--:|:--:|:--:|:--:|:--:|:--:|:---:|
| 2026-06 | 4 | 2 | 4 | 2 | 1 | 13 | — |
| 2026-05 | 3 | 1 | 2 | 1 | 0 | 7 | +86% ↑ |

## 规则效果月度变化
| 指标 | 上月 | 本月 | 趋势 |
|------|:--:|:--:|:---:|
| 有效规则 | 3 | 2 | 📉 -1 |
| 复发规则 | 1 | 2 | 📈 +1 |

## 分类漂移
| 分类 | 上月占比 | 本月占比 | 漂移 |
|------|:------:|:------:|:--:|
| PROC | 43% | 31% | -12% 🟡 |
| ENVR | 28% | 31% | +3% |
```

## 格式原则

- **先结论后过程**: 每个模块先说结果
- **一行一事**: 不堆砌
- **上限折叠**: 超模块上限自动折叠→归档，保持内联≤20行
- **高频高亮**: ≥3次 ⚠️⚠️⚠️
- **收割内聚**: 所有数据在 `harvests/` 内
- **热区定容**: AGENTS.md 错误规则 ≤5条，超限自动降级到温区
- **Git版本化**: harvests/ + AGENTS.md 规则变更自动 commit，支持回滚
- **表格优先**: 归档数据用表格格式，避免 JSON 代码块（减少 token 开销）
- **建议非待办**: 用户自主选择
